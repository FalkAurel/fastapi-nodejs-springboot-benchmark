FastAPI  vs  Node.js / Express  vs  Spring Boot
Vergleich von Antwortzeit und Ressourcenverbrauch unter Last

Ziel des Benchmarks

Dieser Benchmark vergleicht drei populäre Web-Frameworks anhand identischer Aufgaben: eine Datenbank-Abfrage mit Aggregation, Joins und Subqueries wird unter Lastbedingungen (50 gleichzeitige virtuelle Nutzer) ausgeführt. Da alle Implementierungen dieselbe Business-Logik nutzen, misst der Test ausschließlich die Framework- und Laufzeit-Effizienz.

Gemessene Metriken: Antwortzeit (ms)  &  RAM-Verbrauch (MB)

Business-Logik: der /get-age-Endpoint

Jedes Framework exponiert exakt einen HTTP-GET-Endpunkt: /get-age. Dieser führt bei jedem Aufruf dieselbe komplexe SQL-Abfrage aus und gibt das Ergebnis als JSON zurück. Es gibt keinen Cache — jeder Request trifft die Datenbank.

Die SQL-Abfrage im Detail

SELECT
    d.name                                                      AS department,
    AVG(u.age)                                                  AS avg_age,
    AVG(u.salary)                                               AS avg_salary,
    COUNT(u.id)                                                 AS employee_count,
    SUM(CASE WHEN u.salary > da.avg_sal THEN 1 ELSE 0 END)     AS above_avg_salary_count
FROM users u
JOIN departments d ON u.department_id = d.id
JOIN (
    SELECT department_id, AVG(salary) AS avg_sal   -- Subquery
    FROM users
    GROUP BY department_id
) da ON u.department_id = da.department_id
GROUP BY d.id, d.name
ORDER BY avg_age DESC;

Was die Abfrage berechnet

department

Name der Abteilung aus dem JOIN mit der departments-Tabelle.

avg_age

Durchschnittsalter aller Mitarbeiter in dieser Abteilung. Ergebnis wird absteigend sortiert — älteste Abteilung steht oben.

avg_salary

Durchschnittsgehalt aller Mitarbeiter der jeweiligen Abteilung.

employee_count

Anzahl der Mitarbeiter pro Abteilung (immer 1.000, da zyklisch verteilt).

above_avg_salary_count

Anzahl der Mitarbeiter, deren Gehalt über dem Abteilungsdurchschnitt liegt — berechnet über einen korrelierten Subquery-JOIN.

Warum diese Abfrage anspruchsvoll ist

Die Datenbank muss bei jedem Request alle 10.000 User-Zeilen verarbeiten:

Operation Beschreibung
JOIN departments Verbindet jeden User mit seinem Abteilungsnamen
Subquery (Derived Table) Berechnet für jede Abteilung separat den Gehalts-Durchschnitt — ein vollständiger zweiter Tabellenscan
JOIN auf Subquery Verknüpft das Subquery-Ergebnis mit jedem User-Datensatz
CASE WHEN + SUM Zählt zeilenweise, ob Gehalt > Abteilungsdurchschnitt
GROUP BY + ORDER BY Aggregiert 10.000 Zeilen auf 10 Abteilungen, sortiert absteigend

Implementierung je Framework

FastAPI

Asynchroner Endpoint (async def) mit aiomysql-Connection-Pool. Die Query wird non-blocking ausgeführt — der Event Loop blockiert nicht während des Datenbankwartens.

Node.js

Async/await mit mysql2/promise-Pool. pool.execute() gibt ein Promise zurück — ebenfalls non-blocking durch den Node.js-Event-Loop.

Spring Boot

Native Query über ein Spring Data JPA UserRepository. Synchroner Thread-pro-Request-Ansatz mit HikariCP-Connection-Pool. Jeder Request belegt einen Thread bis zur DB-Antwort.

# Dienste stoppen
docker compose down
